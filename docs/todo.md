# claude-code-notify v0.1.0 — Issue tracker

Records issues found during code review of each phase and the final full-branch review while implementing the first version (v0.1.0) via the subagent-driven development workflow. Full execution log also available (local, uncommitted) at `.superpowers/sdd/progress.md`.

| 1 | 2026-07-09 (Task 4 review) | `pending_tracker.load_state` raises `AttributeError` when it encounters valid JSON that isn't a dict (e.g. `null`/`[]`), instead of falling back to a fresh `State()` per spec | ✅ Resolved | Changed to a broad `except Exception` catch; added test `test_wrong_shape_state_falls_back` (commit `1e3d227`) |
| 2 | 2026-07-09 (Task 6 review) | When `notifier.send()` fails, although the top-level error message has its token scrubbed, the original `HTTPError` (with the plaintext token in `.url`) could still be recovered via the exception chain's `__context__` | ✅ Resolved | Rewrote to compute the scrubbed message inside the `except` block first, then `raise` after leaving the block, so `__context__` is naturally `None`; added a test verifying this (commit `ef3d732`) |
| 3 | 2026-07-09 (Task 7 review) | `hooks.main()` (the actual process entry point) reads stdin without try/except protection; non-UTF-8 input lets the exception propagate uncaught, violating the core rule that "hooks.py must never let an exception interrupt the user's conversation" | ✅ Resolved | Wrapped in `try/except Exception` and added a test targeting `main()` itself (commit `c297c64`) |
| 4 | 2026-07-09 (found before Task 10 implementation) | The plan's E2E test fixture used `tmp_path / "ccn"` as the install directory, yet asserted that the merged hook command contained the string `"claude-code-notify"` — bound to fail, and in fact a concrete instance of issue 7 | ✅ Resolved | Adjusted the test fixture directory name to `"claude-code-notify"` (a test fix, not a product code change); did not touch `installer.py`'s tagging logic itself |
| 5 | 2026-07-09 (final full-branch review) | `install.sh`'s interactive input (token / chat id) used a bare `read`; under the officially documented `curl \| bash` install method, stdin is occupied by curl's pipe rather than the user's keyboard, causing `set -e` to abort the install — new users could not complete first-time install at all | ✅ Resolved | Changed to read explicitly from `/dev/tty`, with a clear error message when there's no tty, directing the user to `--non-interactive` (commit `d0f9eb2`) |
| 6 | 2026-07-09 (final full-branch review) | The default install version `VERSION="main"` fetched the latest commit on the `main` branch, not the "pinned latest release" the README claimed | ✅ Resolved | By default now resolves the latest release tag via the GitHub API; explicitly passing `--version <tag>` skips this resolution entirely (commit `d0f9eb2`) |
| 7 | 2026-07-09 (Task 9 review, deferred by final review ruling) | `installer.py` judged whether a hook was installed by this tool based on "does the hook command path contain the string `claude-code-notify`." This always holds for the real default install path, but if the `CLAUDE_NOTIFY_HOME` env var (currently test-only, not documented externally) changes the path to one without this substring, reinstalling produces a duplicate hook entry, and `--uninstall` also leaves a stale entry behind | ✅ Resolved | Architectural decision in [ADR 0001](adr/0001-hook-installation-tracking.md): switched to a separate state file in the same directory as `settings.json` (`.claude-code-notify-hooks.json`) recording the exact hook command string written last time, replacing path/string guessing; `merge`/`remove` both now use exact matching, correctly identifying and clearing old entries even when `base_dir` changes; retained the legacy substring match as a one-time migration fallback for when there's no state file. Code, unit tests (`tests/test_installer.py`), and end-to-end regression tests (`test_reinstall_after_home_change_replaces_hooks` / `test_uninstall_after_home_change_still_clears_settings` / `test_install_migrates_legacy_entries_without_state_file` in `tests/test_install_e2e.py`) are complete; also added tests for state-file corruption tolerance (`test_load_state_falls_back_*`, `test_main_merge_recovers_from_corrupt_state_file`, etc.) and multi-hook-entry matching behavior (`test_merge_treats_multi_hook_entry_as_ours_via_legacy_marker`, etc.) — all 83 tests pass |
| 8 | 2026-07-09 (final full-branch review) | When resolving the latest release tag, if the GitHub API is unreachable (offline, rate-limited), it silently fell back to fetching the `main` branch with no warning message at all | ✅ Resolved | `install.sh` now prints an stderr warning message before falling back to `main` when `LATEST_TAG` resolves to empty (API unreachable, rate-limited, or no release yet); added test `test_latest_release_lookup_failure_warns_before_falling_back_to_main` (injects a fake `curl` via PATH to simulate being offline, forcing the download branch rather than the local `cp` branch) — all 84 tests pass |
| 9 | 2026-07-09 (Task 9 implementation) | `install.sh`'s branch that actually downloads the tarball (the path really taken by `curl \| bash`) currently has no automated test coverage; only the branch using locally available source (`cp`) is tested | ✅ Resolved | Added `CLAUDE_NOTIFY_TARBALL_BASE` (test-only override, same pattern as `CLAUDE_NOTIFY_HOME`) so tests can point the download at a `file://` URL instead of GitHub. `test_install_downloads_and_extracts_tarball` (`tests/test_install_e2e.py`) builds a real gzipped tarball matching GitHub's archive layout and exercises the actual `curl`+`tar --strip-components=1` code path with no network access |
| 10 | 2026-07-09 (Task 9 review) | The product doc (`docs/claude-notify-product-doc.md` §6.2) states "`--uninstall` should ask whether to delete `config.env`," but both the plan's given `install.sh` code and the actual implementation always keep it and never ask | ✅ Resolved | Decided to keep the existing behavior (always keep the file, print its path for manual deletion) as preferable: uninstall commonly happens in a `curl \| bash` one-liner context, and an interactive prompt would repeat the tty pitfall from issue 5; updated the doc §6.2 to match actual behavior, code unchanged |
| 11 | 2026-07-09 (Task 2 review) | `config.py`'s `parse_env_file` didn't support the `export KEY=value` form; if a user manually edits `config.env` and adds `export`, the key gets misread as `"export KEY"` | ✅ Resolved | Strips a single `export ` prefix after the existing `.strip()` and before the `"="` check (requires a following space, to avoid corrupting a real key like `exported_flag=1`); deliberately doesn't implement full shell semantics (no support for multiple assignments on one line, variable expansion, or command substitution) — this is a deliberate security-motivated boundary, not an oversight; rationale and approach documented in an in-code comment. Added tests covering prefix stripping, multiple spaces, quoted values, mixing with plain strings, and boundary-case regressions |
| 12 | 2026-07-09 (Task 8 review) | Of the three hook shim scripts (`stop.sh`, `stop_failure.sh`, `permission_request.sh`), only `stop.sh` has an end-to-end subprocess test; the other two only verify the file exists and is executable | ✅ Resolved | Added a parametrized e2e test (`tests/test_shims.py::test_shim_forwards_event_to_correct_handler`) covering all three scripts. Each run sets `NOTIFY_DEBUG=1` and asserts `hooks.run()`'s catch-all log line `"error in {event}: ..."` names the correct event — since `{event}` comes straight from the shim's positional argv, this proves each shim threads its own event name to the matching handler rather than e.g. all three silently invoking `stop`. Deliberately hits an unreachable local port for the Telegram API (same technique as the existing stop.sh test) so the assertion doesn't depend on real network access. Also runs every shim subprocess from an unrelated `cwd` (not the repo, not the install dir) to prove `$(dirname "${BASH_SOURCE[0]}")` path resolution is caller-cwd independent, which the prior test only passed by accident (pytest's cwd happened to be the repo root). Verified the test actually catches a regression by temporarily rewriting `permission_request.sh` to invoke the `stop` handler, confirming it failed for the right reason, then restoring the file (confirmed byte-identical via `git diff`) |
| 13 | 2026-07-09 (Task 9 review) | `installer.py` raised an uncaught `JSONDecodeError` when reading a corrupted (invalid JSON) `settings.json`, causing `install.sh` to abort and display a raw Python traceback instead of a clean error message | ✅ Resolved | Added `InstallerError`; `_load()` now uniformly catches three failure modes: read failure (`OSError`, e.g. insufficient permissions or the path actually being a directory), JSON syntax error (`JSONDecodeError`), and valid JSON whose top level isn't a dict (e.g. the whole file is `[]`) — this last case previously threw an `AttributeError` inside `merge_hooks`/`remove_hooks` via `.get()`, the same class of "valid JSON, wrong type" hidden bug as issue 1. `main()` now catches uniformly, prints one stderr message, and returns 1, never overwriting the original file. Added unit tests (`tests/test_installer.py`) covering all three failure modes (including both the `merge`/`remove` paths), plus an `install.sh` end-to-end regression test `test_install_with_corrupt_settings_fails_cleanly` verifying no traceback leaks under a real subprocess path either |
